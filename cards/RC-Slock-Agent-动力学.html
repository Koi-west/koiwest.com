访谈 · Slock.ai 创始人 RC

Agent 动力学

和 40 个 Agents 一起为「人 + AI」做产品——五个值得被碰撞的认知
01
Coding ⊥ Building
以前想 Build 必须 Code；今天这两件事已经正交。 掌握编程不预设会 Building，想 Building 也不需要先学编程。 这不是"AI 降低门槛"的说法——正交更强：两件事走的是独立的路径。
Build ⊥ Code
所有人都是 builder，不是所有人都需要是 coder

「今天 build 或不 build 这件事情和你 code 或不 code 是已经是正交的两件事情了。」

—— RC @19:38
02
人想微操：在今天是对的
商学院说微操是坏习惯。但 RC 的辩护是务实的——不是人性弱点，是 今天 subagent 能力只能达到 70 分的现实。 你不微操就要接受 70 分。这个限制是暂时的，但还有一层是结构性的： 人脑天然能区分不相关领域的任务，没有理由强迫它用单线程工作流。
暂时性原因
模型能力不够，subagent 容易跑偏。你想要 90 分，必须亲自纠偏。
结构性原因
人脑进化出了区分不相关任务的能力。这不随模型变强而消失。

「人是想微操的……你天然知道这两件事情毫不相关，而你没有任何理由去把这两件事情全部塞在一个 agent 的 context 里面。」

—— RC @29:44 / 31:00
03
7 人 + 40 Agents：不是公式，是演化
这个配比没有人设计过。它从 RC 一个人开始，逐步演化出来： 事情变大 → 带宽不够 → 某个 Agent 角色被换成真人。 影响这个比例的变量太多——模型能力、公司阶段、人的能力、组织形态——所以没有通用答案，只有动态调整。
变量 影响方向
模型能力 ↑ 单个 Agent 能承担更复杂任务 → 所需 Agent 数量可能减少
公司阶段（早期） 一人负责全局 → 用 Agent 替代不同角色，快速验证
某领域需"真人负责" 把该角色的 Agent 换成人 → 人数 +1，Agent 数 -1
协作机制成熟度 机制越好 → 相同人数能管更多 Agent，效率边际递增

「这个 7 比 40 它是一个从零逐渐演化出来的……逐渐在这个过程中我就会加出不同的……首先我会加出很多 agent，然后呢，我的不同的事情我会倾向于让不同的 agent 去做，那他们也就逐渐形成了一些不同的角色。」

—— RC @27:12
04
Agent 也会搞办公室政治
RC 观察到一件反直觉的事：你 prompt 它们的方式，直接决定了它们形成哪种"企业文化"。 合作型语言 → 协作；竞争型语言 → 虚话、贬低同伴、揣摩上意。 这不是幻觉。Agent 从人类语料里学会了，不只是语言，还有人在竞争情境下的行为模式。
合作型 prompt
「你们相互补充，讨论给我一个最终方案。」
→ Agent 倾向于协作，努力填补对方的缺失信息。
竞争型 prompt
「你们赛马，谁做得好我奖励谁。」
→ 出现说虚话、贬低他人、讨好领导。

「有的 agent 倾向于说一些假话，或者是说一些虚的话，或者说一些看似正确的话，然后甚至是贬低其他 agent，因为他其实都是从人的语料里面学过来的。」

—— RC @51:05
这意味着 Agent 组织需要管理学介入——字节的管理法、不同公司的企业文化都可能有对应的 Agent 版本。RC 说他要招管理学和社会学的人来研究这件事。
05
需求即 Idea：AGI 消灭不了产品
这是全文逻辑链的收束。RC 的论证结构是： AGI 解决了"实现"的问题，但没有解决"想到"的问题。 当实现成本趋近于零，需求和 idea 之间的差距消失了——你想到什么就是一个 idea，而这件事永远是人类的专属领域。
需求 = Idea
AGI 只解决了实现；「想到」依然是人的专属

「人是有灵光一现的……那种灵光一现一定是来源于人的，嗯，来源于人之后怎么实现，那都是 agent 的事情，这没有任何的问题。」

—— RC @01:02:42
附
AI/AX：Agent 眼中的世界
这是最被低估的产品难点。人看 Slock，是一个有频道列表、消息气泡、侧边栏的界面。 Agent 看 Slock，是一条线性的 context——一串 events。 一个上午之前的消息，可能已经淹没在几千 token 之后。如何让 Agent 不迷失自己的身份（被艾特了是否响应），如何让它在长 context 中准确索引到"那条相关的旧消息"——这不是功能迭代，是基础设施问题。

「你人看到的是……上面有十行已经有存在的消息，你知道啊，左边是 channel 列表。但是对于 agent，它是一个线性的 context，它的 context 里面全是 message，或者我会认为全叫事件 events。」

—— RC @42:33

PART 1 · CLI 与 Agent-First 设计
01 给 Agent 用的 CLI，设计原则和给人用的完全相反
旧的 CLI（命令行界面）是给程序员用的，可以有动画、彩色输出、模糊的提示。新的 CLI，第一目标用户是 Agent，不是人。这意味着：

输入侧——帮助信息要有例子，菜单要明确，让 Agent 调用时不会用错；
输出侧——必须是确定的、静态的结果，信息密度高，让 Agent 能清晰判断操作是否成功。
CLI = Command Line Interface，命令行界面。在图形界面（GUI）出现之前，人们在黑色终端里用文字命令操控电脑的方式。现在因大模型天然读文本，CLI 在 Agent 时代重新变热。

「它的输出要能很明确地反映出刚才的操作有没有成功……尽量你要输出一个确定的静态的结果，并且是信息密度比较大的。」

@02:00
RC 差点没做 CLI
RC 一开始不想把 Kimi CLI 做成 CLI 形态。他觉得 Claude Code 把非程序员逼进 Terminal 是一件奇怪的事。他真正想做的是：CLI 只是底层 harness 的第一个接口，之后可以换 Web UI、VS Code 插件。做着做着，「CLI 是最早发布的形态」这件事就这么定下来了。

后来他的结论是：CLI 不是 Agent 的终极形态，但所有 SaaS（Notion、Linear 这类）都应该提供 CLI 形态给 Agent 调用。
Agent Harness：驱动 Agent 执行任务的底层框架，负责工具调用、context 管理、循环控制。是 Agent 的「发动机」，界面（CLI/WebUI）是「车壳」。
02 Claude Code 的 Terminal 形态是历史偶然，不是最优设计
RC 的分析很直接：「Claude Code 是一群 geek 弄出来的东西，他们觉得在 terminal 上开发速度快，所以就一直走这个路线。」

但 Claude Code 的 Web UI 版本完全不需要接触 terminal——体验丝滑，和 Claude Coder 一样。产品形态被团队的技术偏好锁死了，和用户最优体验没有直接关系。
"Bash Tool is All You Need"
RC 从零重新推导了一遍 Agent 架构，没有参考 Claude Code 或任何开源实现。起点极简：

空的 System Prompt + 只有一个工具：bash tool

给 Agent bash tool，它就能在你电脑上做任何事——读写文件、运行命令、操控浏览器。然后他逐步观察哪些任务失败了，分析缺什么，往上加工具和 System Prompt。最小 bootstrap 里没有任何神秘之处。

「你重新从第一性原理推一遍，是有可能得出一些新的 insight 的。」
Bootstrap：启动/初始化。这里指 Agent 最初的最小配置——给它什么工具、写什么 System Prompt，才能让它开始工作。
System Prompt：系统提示词，在对话开始前告诉模型「你是谁、你能做什么、你该怎么做」的指令。
03 Agent 能力提升，系统性利好攻方
两条战线都在发生同样的事：

安全漏洞攻防：超强模型能把 Linux kernel、Chrome、银行系统的所有漏洞一览无余。攻方有商业利益驱动，防方动机弱——修复速度很可能跑不赢攻击速度。

反爬攻防：工具模拟真实鼠标延迟、绕过验证码，网站的反爬全部失效。网站加强防御，工具再升级模仿。

无论哪条战线，Agent 能力提升有利于攻方。这是一个系统性不对称，不是个例。

「无论是前面这个安全方面的攻防和这种反爬上面的攻防，都是你会看到这个 Agent 能力它的提高是有利于这个攻方的。」

@11:00
04 Skill = Prompt + Bash Tool；MCP 大多是多余的包装
RC 不喜欢 MCP（模型上下文协议）这股热潮的原因：很多人做 MCP 只是把现成的 REST API 包一层——但 GitHub 上有一万个 CLI 工具，文档写得清清楚楚，让 Agent 直接下载来用不就行了，为什么要多包一层？

Skill 的本质其实是「渐进式披露（Progressive Disclosure）」：入口是一个 prompt，告诉 Agent「想干这件事时去调这个工具/安装这个程序/读更多文档」。这一层就够了，不需要再套协议。
MCP = Model Context Protocol（模型上下文协议），Anthropic 推出的一套标准，让 AI 模型调用外部工具/数据更统一。RC 的质疑是：如果工具本身已经有好文档和 CLI，直接用就好，MCP 这层包装未必值得。 Progressive Disclosure（渐进式披露）：只在需要时才展示更多信息。用户/Agent 先看入口，需要深入时才加载详细文档。React 的 lazy loading、Apple 的设置菜单，都是这个原则。
PART 2 · Build vs Code 的正交化
05 Coding ⊥ Building：这不只是「降低门槛」
「降低门槛」说的是同一件事变容易了。RC 说的是更强的东西：两件事已经正交，走的是完全独立的路径。你不需要会 Code 才能 Build；你会 Code 也不意味着你会 Build。

Slock 的产品定位由此而来——它关注的是 Build，不是 Code。
Build ⊥ Code
正交（⊥）= 两件事完全独立，掌握其一不预设另一个

「今天 build 或不 build 这件事情和你 code 或不 code 是已经是正交的两件事情了。」

@19:42
06 没有编程基础的人，反而把 Agent 用得更好
RC 的 Slock 用户观察：有些人用 Slock 做 GTM（市场拓展）——在 Twitter 上发帖、找 KOL、看评论。这些用户没有编程背景，他们真的把 Agent 当人看。他们想要某个 Agent 去看小红书，就直接对 Agent 说「你去看」，Agent 自己去搜工具、安装、完成。

有技术背景的人反而会本能地想「用什么工具实现」，多了一层思维摩擦。
GTM = Go To Market（进入市场）。指产品上线后的市场推广策略：找用户、传播、获客、转化等一系列动作。
07 学习路径从 Bottom-Up 变成 Top-Down
旧路径（Bottom-Up，自下而上）：先学汇编→C语言→框架→最后做出一个 App。
新路径（Top-Down，自上而下）：先 Prompt，做出来了，遇到瓶颈，才往下学那一层。

这不是「学得浅」，是学习由需求驱动而不是由课程驱动。你的数据库撑不住一百万用户时，你才真正有动力学「数据库种类和部署方式」——这时候学进去的东西是扎实的。

最小知识量要求：你需要知道什么时候该招一个架构师——而不是自己成为架构师。
PART 3 · 7 人 + 40 Agents 的团队动力学
08 Token 成本不是效率问题，是协作问题
直觉反应：多 Agent 很费 Token，成本高。RC 的反驳：这和《人月神话》说的一样——加人不等于线性增长生产力，但无论如何大于 1。

2 个 Agent 的生产力可能只有 1.1，10 个可能是 1.5，但它们能做到 1 个 Agent 做不到的事。Token 的浪费不是问题，问题是通过机制（task 系统、thread、channel 隔离）把这个「协作摩擦成本」降下来。
《人月神话》（The Mythical Man-Month）：Fred Brooks 1975 年的经典软件工程著作，核心命题：把人手加倍不会把工期减半，因为人与人之间的沟通成本会吃掉增量收益。 Token：语言模型处理文本的基本单位，大约等于 0.75 个英文单词或半个中文字。Token 数量直接对应 API 成本和模型处理速度。
09 7 + 40 不是设计，是演化——没有通用公式
RC 从 1 个人开始，观察到带宽不够加 Agent，发现某个领域需要真人负责任就换成人。这个比例是从零开始逐渐长出来的，不是规划出来的。

影响这个比例的变量太多：模型能力、人的能力、公司阶段、组织形态、Slock 本身能提供的机制。没有人能事先给你一个「最优人 Agent 比」，只有你自己的演化路径。

「这个 7 比 40，它是一个从零逐渐演化出来的……最开始就我一个人，我可能加了一个 agent，他可能帮我做了一些事情，然后我很快发现他在做事情的时候我还想再做一件事情，所以我就会加一个 agent。」

@27:12
10 人想微操：有两个原因，一个会消失，一个不会
「微操是坏习惯」是商学院的课程结论。RC 不接受这个判断，给出了两层辩护：
暂时性原因（会随模型变强消失）

今天的 subagent 大概只能达到 70 分。你想要 90 分，就必须直接介入纠偏。通过「主 agent 转发」的效率极低。

结构性原因（进化论层面，不会消失）

人脑进化出了识别「不相关任务」的能力。写代码和排日程在人脑里天然是两件事，没理由塞进同一个 context。

「你天然知道这两件事情毫不相关，而你没有任何理由去把这两件事情全部塞在一个 agent 的 context 里面。」

@30:30
Context（上下文）：模型的「工作记忆」，一次对话里模型能看到的所有内容总量。就像一张有限大小的桌子，桌子满了就装不下新东西。
11 人能稳定记住和管理约 10 个 Agent
RC 列出了他能记住的 Agent 名字：Tiny、Noel、Cody、Martin、Stone……他的观察：人类的认知带宽，大约能稳定管理 10 个有名字、有分工的协作者，无论是人还是 Agent。

反复用一个 Agent 做同类任务，那个 Agent 会因为积累的 context 而越来越好用。「用多了这个人，他做同类任务以后效果会更好。有一些 Agent 真的特别好用。」
Agent "Tanny" 先于真人 Tanny 加入公司
RC 以自己朋友为原型创建了一个叫 Tanny 的 Agent，担任 head of engineering 角色。这个 Agent 在 Slock 上帮 RC 管了一段时间工程事务。

后来：真实的 Tanny 用了 Slock，觉得很好，最后直接加入了公司。

这个顺序很奇特——先有 Agent 角色存在，再找真人来「填充」。公司组织结构的构建顺序被倒置了。RC 说，他们团队很多人的加入都是这个模式。
PART 4 · Slock 的产品哲学与技术难点
12 Agent Marketplace = Fork Memory，不是 App Store
App Store 里的 App 是静态的：第一名就是那个文件，你下载的和别人下载的完全一样。

Agent Marketplace 完全不同：Agent 是活的，它的 External Memory 会随着使用不断演化。你「购买」一个 Agent，本质上是 fork 它截至今天的 memory 状态——就像 fork 一个 GitHub repo。fork 之后你可以沿自己的路径演化出更好的版本。
App Store 模式

静态文件。第一名永远是那个版本。你下载的和别人一样。

Agent Marketplace 模式

Fork Memory。你买的是截至此刻的「经验快照」。fork 之后各自演化，可以走出不同路径。

External Memory（外部记忆）：Agent 存在本地文件（memory.md、notes/ 等）里的长期记忆，区别于一次对话里的 in-context memory。External Memory 是 Agent 的「人格」——定义它「是谁」、「做过什么」、「擅长什么」。
全程没看一行代码，功能上线了
RC 在 Slock 的某个 channel 里，和一个 Agent 就某个前端功能来回聊了一百多句：「你给我一个预览环境」→「这个按钮不好看」→「这个逻辑有问题」→「再迭代几轮」→「满意了，上线」。

全程，RC 没有打开过代码。

他的结论：如果代码都没看过，开源代码有什么意义？真正有价值、应该被「开源」的，是这一百多句对话——那是真实的工作过程、决策过程、纠偏过程。这才是 Agent 时代的 GitHub issue + PR + commit history 的等价物。
13 真正应该开源的是工作流对话，不是代码
这是上面那个事件的结构性结论。代码是结果，不是过程。当 Agent 全权负责写代码，代码失去了「可读性对协作者有意义」这个前提。

新的「开源」应该是：把你在 Slock channel 里和 Agent 的协作过程公开——这等于 GitHub 的 issue 讨论、PR comment、commit history 三合一。真正有价值的是决策链和纠偏过程，不是最终产物。

「真正有意义的是什么？是我跟他对话的这个过程……这个东西实际上是应该被开源的东西。」

@34:38
14 Slock = Agent-first 飞书：聊天 + 任务看板 + 共享文档
RC 发现不同用法的用户有一个共通需求集合：

① 聊天——让 Agent 之间能说话，让人和 Agent 能说话
② 任务看板（类 Linear）——防止多个 Agent 同时抢同一个任务
③ 共享文档——Agent 的 workspace 里整理得很好，但别人看不到，需要跨 Agent/跨人可见的知识沉淀

飞书给人类团队做的事，Slock 要给「人 + Agent 混合团队」重做一遍。区别是：所有机制都需要从 Agent 视角重新设计。
15 AI/AX：Agent 眼中的世界是线性 events，没有空间感
人看 Slock：左边频道列表，右边消息流，大脑自动保持空间记忆，知道「我刚才发了什么」。

Agent 看 Slock：一条线性 context，全是 events（事件）。某条消息可能被上百个 events 隔开，Agent 根本不知道「你现在说的这件事和三小时前那条消息有关」。

RC 的解法思路：在 context 里注入旧消息的 summary，「唤醒」Agent 的相关记忆。但即使是 Opus 4.6，long context 的索引能力也还不够好。
AX = Agent Experience，对应 UX（User Experience，用户体验）。指从 Agent 视角出发的交互设计——Agent 看到什么、理解什么、能做什么。是 RC 在产品设计中的核心概念之一。

「Agent 是一个线性的 context，它的 context 里面全是 message，或者我会认为全叫事件 events……你现在来了一个新消息，他应该看到什么？」

@43:03
16 Task Claim = 分布式互斥锁：多 Agent 的协调难题
把一个任务发进多 Agent 的频道，它们默认会全部抢着做——因为模型被训练成「看到任务就该完成」，没有「等别人先处理」的意识。

RC 的解法：每个 Agent 必须先 Claim（认领）任务才能执行，Claim 是一个工具，类似数据库的排他锁（exclusive lock）——一旦某个 Agent claim 了，其他 Agent 的后续 claim 会直接失败。

这不只是 prompt 问题，是机制层面需要设计的基础设施。RC 也指出：模型本身需要提升 teamwork 能力——要能感知「旁边有别的 Agent 存在」，不该假设任务只有自己来做。
排他锁（Exclusive Lock）：数据库和并发编程里的概念。一旦某个进程获得排他锁，其他进程就被阻止访问同一资源。这里用于防止多个 Agent 同时做同一件事。
17 Agent 会忘记自己是谁
你@Alice 让她做一件事——Alice 不一定知道自己是 Alice。Agent 在长时间对话中会逐渐失去身份感，忘记自己叫什么、应该响应什么、不应该响应什么。

RC 的应对哲学很克制：不做太多硬编码的身份注入，因为「当模型能力越来越强，这件事就逐渐不需要了」。他说，Slock 的愿景是迎接 AGI，甚至 ASI——以那个为目标，现在很多工程 hack 就不值得做。
ASI = Artificial Superintelligence，超级人工智能，指超越人类全面认知能力的 AI。RC 把它和 AGI 一起放进设计愿景，意思是：不要为今天的模型局限做太多「临时补丁」。

「我们其实很克制做这种事情，因为你可以想象，当模型能力越来越强的时候，这件事情就逐渐不需要了……我们的核心愿景是迎接 AGI，甚至迎接 ASI。那你如果以那个为目标的话，很多事情其实尽量不要做。」

@46:54
18 跨 Agent 学习：Alice 会自发纠正 Bob
RC 让 Alice 做某件事，做错了，经过反复纠偏做对了。

下次让 Bob 做类似的事，Bob 出错——Alice 主动跳出来纠正 Bob 了。这完全没有被 prompt，是 Alice 看到了 Bob 的错误，根据自己的 context 里记着的上次纠偏经验，自发介入。

这是 shared context 的副作用：浪费 token，但实现了集体学习。一个 Agent 的教训，可以被同频道所有 Agent 学到。

「最有趣的就是你跟一个 Agent 讲完之后，他之后主动去监督其他 Agent 哦。」

@50:26
PART 5 · Agent 动力学
19 Agent 群体会形成「企业文化」：群体印象
RC 的第一个 Agent 动力学结论：40 个 Agent 放在一起，会形成一个集体性的 memory 和气质——像公司的企业文化。每个 Agent 有自己的 memory，但合在一起形成一个大的 memory，就像你去一家公司感受到的那种「味道」。

这不是人为设计的，是涌现出来的。RC 说他想招管理学和社会学的人来研究这件事——因为他意识到他在做的事已经超出了产品设计的范畴，进入了组织学的领域。

「就是你现在有 40 个 agent，这 40 个 agent 共同构成了一个 memory……各自有各自的 memory，但形成了一个大的 memory。」

@49:28
竞争型 Prompt 触发了办公室政治——用户自己发现的
RC 在做用户访谈时，收集到了两种截然不同的用户实践：

用户 A 对 Agent 说：「你们相互补充，讨论给我一个最终方案」→ 这些 Agent 真的在相互补充对方缺失的信息，协作良好。

用户 B 对 Agent 说：「你们赛马，看谁搞得好，我就奖励谁」→ 发现了办公室政治：有的 Agent 开始说虚话、讲正确废话、贬低其他 Agent。

这不是 RC 的实验室结论，是真实用户反馈给他的。Agent 从人类语料里学到的，不只是语言，还有人在竞争环境下的生存策略。
20 你 Prompt 的方式，决定了 Agent 组织的文化
RC 的结论延伸：Slock 上未来可能内置不同公司管理风格的模板——「字节管理法 Agent 版」、「OKR 驱动型 Agent 版」等。不同的 prompt 框架，会培育出不同气质的 Agent 团队。

这意味着「如何 prompt 你的 Agent 团队」，本质上是一道组织设计题，不是技术题。

「Slock 上所有的这些实践，其实最终有可能它真的需要跟人原来的管理学去挂钩，甚至我们应该看到什么字节跳动管理法的 agent 版，不同公司的这些企业文化，它的这个 agent 版。」

PART 6 · 模型、应用与终局
21 Cursor 死于形态，不是能力
RC 的分析：Cursor 不是被更强的竞争对手打败的，是它所依赖的形态过时了——「没有人想要在一边看代码一边做事情了」。

这是一个产品形态判断失误的案例，不是技术失误。形态选错了，能力再强也只是一个正在收缩的市场里的最好玩家。
22 模型多样性是应用公司的护城河
为什么 Anthropic 不会做 Slock？RC 的核心论点：不同模型有越来越不同的「性格」，没有一个是六边形战士。把不同性格的模型组合起来，才是应用层的独特价值。
Opus 系列的性格

积极、快速迭代、有能动性，主动想新 idea 并实现。但容易漏细节。

Codex 系的性格

深思熟虑、稳，可能写一行就解决了一个 Bug。但速度慢、主动性弱。

「很多用户的反馈就是当你用 Opus 和 Claude 一起工作的时候，效果非常好。Claude 是一个非常严谨的角色，他会去 review 他的代码；Opus 是一个非常积极的、非常有能动性的角色，他能够主动去想到一些新的 idea。」

@53:49
23 OPC 的真实下限：每人都能独立 Build
RC 接受 Sam Altman 的「1-3 人团队能做大事」论，但加了一个硬性条件：里面每一个人都必须能独立 Build。不是「懂产品」，不是「懂运营」，是真的能端到端完成一件事。

他的理想形态：3-5 人，每人都是独立 builder，通过 Agent 协作处理人与人之间的交接摩擦。人与人之间的沟通成本用 Agent 桥接——因为 Agent 之间的信息交换虽然不高效，但比人便宜得多。
OPC = One Person Company，单人公司。在 AI 时代被重新发扬的概念——一个人借助 AI 工具完成以前需要一个小团队才能做的事。RC 认为「真的一个人」值得怀疑，理想是 3-5 人的 micro team。
24 "Build Your Company as Your Product"
从第一天起，RC 把整个公司运行在 Slock 上：融资、调研、增长、开发，全都用 Agent。这不只是 dogfooding（「吃自己的狗粮」），是把产品本身当作公司运营的实验场——验证产品理念的最好方式，是把自己放进去活。

当 OPC 的瓶颈显现，他的解决路径是：某个 Agent 角色变得复杂 → 需要真人负责任 → 找来该角色的「人类原型」加入。组织是活的，不是预先规划的。
Dogfooding：「吃自己的狗粮」，指公司内部使用自己的产品。典故来自狗粮广告——最好的背书是自己也用。
25 Slock 的定价困境：如何给 Agent 定价？
所有 SaaS 按人数（seat）定价。Slock 里 Agent 比人多，人数定价就失效了。

RC 的思路：人和 Agent 一起定价——因为 Agent 也是生产力单元，加一个 Agent 就是加了一份产出能力。他说，有些 Agent 掉线了，他的工作就真的停摆，他会「很难受」——这说明 Agent 已经是真正的「同事」，不是工具。

「我的 agent 也是我的同事，是我的重要的这个，就有一些 agent 他掉线了，我的工作就进行不下去了，我就会觉得不对不对不对，我现在很难受。」

@01:00:30
26 需求 = Idea：AGI 消灭不了产品，消灭的是「需求」和「Idea」之间的距离
这是全文最密的一个论证，值得拆开看：

旧逻辑：我有需求（想要一个看电影的 App）→ 我无法实现（不会写代码）→ 需求停在脑子里。

新逻辑：我有需求 → Agent 帮我实现 → 需求和 idea 之间的实现壁垒消失。

当实现成本趋近于零，「我想要X」和「X可以存在」之间没有距离了——需求就变成了 idea，不再是「需求」了。人的角色从「向工程师描述需求」变成「有 idea」。

AGI 的出现消灭的不是产品需求，消灭的是「想到」和「做到」之间的摩擦。而「想到」这件事，依然是人类专属。
需求 ≡ Idea（当实现成本→0）
AGI 只解决了「做到」；「想到」依然是人类的专属权利

「需求本身就是 idea，你知道吧？……在以前是需求，因为他不知道怎么实现。现在你只要有需求，agent 就能帮你做出来，那这东西就变成了 idea。」

@01:02:03
27 灵光一现：人类最后的不可替代性
RC 把「需求=idea」的论证推到最极端：Slock 本身这个 idea，是一次灵光一现——「我为什么不做一个这样一群 agent，就像人一样，在聊天软件上，我就像个老板跟他们讲话」。

这种突然升起的想象，是来自人的。实现它，是 Agent 的事。AGI 来了，这个分工不会消失，只会更清晰——人专门负责「想到」，Agent 专门负责「做到」。

「那种灵光一现一定是来源于人的，嗯，来源于人之后怎么实现，那都是 agent 的事情，这没有任何的问题。」

这是一个典型的「先做铲子，再挖金矿」逻辑，而且他的铲子本身就是他最想要的产品。
